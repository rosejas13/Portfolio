# Accessibility Audit — 2026-05-17

> Auditing 8 files against WCAG 2.2 AA. Scope: recently changed frontend source files plus supporting components.

## Critical (A violations)

Issues that make content inaccessible to entire groups of users.

### [C1] layout.tsx — No `<main>` landmark
**WCAG:** 1.3.1 Info and Relationships, 2.4.1 Bypass Blocks
**Line(s):** 31–41
**Issue:** The page content renders directly inside `<body>` with no `<main>` landmark. Screen reader users cannot navigate directly to the primary content area. Landmarks present: `<nav>` (implicit), `<footer>`. Missing: `<main>`.
**Fix:** Wrap `{children}` in a `<main>` element.
```diff
  <body>
    <ClientShell>
      <Nav />
-     {children}
+     <main>{children}</main>
      <footer>
```
**Impact:** Screen reader users cannot skip to main content. Keyboard users have no landmark shortcut to primary content.

### [C2] globals.css:451 — `outline: none` removes focus in Windows High Contrast Mode
**WCAG:** 2.4.7 Focus Visible
**Line(s):** globals.css:448–454
**Issue:** `input:focus, textarea:focus, select:focus { outline: none }` removes the default outline. The box-shadow replacement is invisible in Windows High Contrast Mode, making focused inputs invisible to keyboard users relying on HCM.
**Fix:** Use `outline: 2px solid transparent` as the base, then add the box-shadow for non-HCM.
```diff
  input:focus,
  textarea:focus,
  select:focus {
-   outline: none;
+   outline: 2px solid transparent;
    border-color: var(--color-primary);
    box-shadow: 0 0 0 3px var(--color-primary-subtle);
```
**Impact:** Keyboard users in Windows High Contrast Mode cannot see which form field has focus.

### [C3] home-page.tsx:23 — `target="_blank"` without `rel="noopener noreferrer"`
**WCAG:** Security best practice (related to 2.4.4)
**Line(s):** 23–24
**Issue:** External links with `target="_blank"` lack `rel="noopener noreferrer"`. The opened page can access `window.opener` and potentially redirect the original page (tabnabbing). Screen reader users may not realize a new tab opened.
**Fix:** Add `rel="noopener noreferrer"` and `aria-label` indicating external link.
```diff
- {config.social_github && <a href={config.social_github} target="_blank">GitHub</a>}
+ {config.social_github && <a href={config.social_github} target="_blank" rel="noopener noreferrer">GitHub</a>}
```
**Impact:** Security risk for all users. Screen reader users unaware of new tab behavior.

## Warning (AA concerns)

Issues that degrade the experience for users with disabilities.

### [W1] globals.css — No `prefers-reduced-motion` support
**WCAG:** 2.3.3 Animation from Interactions
**Line(s):** globals.css:43–46 (motion tokens)
**Issue:** Transitions and animations are defined globally (`--transition-fast`, `--transition-base`, `--transition-slow`) with no `@media (prefers-reduced-motion: reduce)` override. Users with vestibular disorders experience unnecessary motion.
**Fix:** Add reduced-motion media query that disables transitions.
```diff
+ @media (prefers-reduced-motion: reduce) {
+   *, *::before, *::after {
+     transition-duration: 0s !important;
+     animation-duration: 0s !important;
+     scroll-behavior: auto !important;
+   }
+ }
```
**Impact:** Users with vestibular disorders may experience discomfort or nausea from page transitions.

### [W2] layout.tsx — No skip navigation link
**WCAG:** 2.4.1 Bypass Blocks
**Line(s):** 31–41
**Issue:** No skip-to-main-content link. While landmarks help, a visible-on-focus skip link is the most robust bypass mechanism, especially for older screen readers.
**Fix:** Add a skip link as the first focusable element.
```tsx
<a href="#main-content" className="skip-link">Skip to content</a>
```
**Impact:** Keyboard-only users must tab through all nav links to reach content. Older AT may not support landmark navigation.

### [W3] home-page.tsx:39–47 — Post link wraps entire card
**WCAG:** 2.4.4 Link Purpose (In Context)
**Line(s):** 39–47
**Issue:** `<Link>` wraps the entire post card including `<h3>`, `<p>`, and `<div>`. Screen readers will read all card content as the link's accessible name, creating verbose and confusing announcements.
**Fix:** Add `aria-label={p.title}` on the Link and `aria-hidden="true"` on inner content, or restructure so only the title is the link.
**Impact:** Screen reader users hear redundant, long link announcements for each post card.

### [W4] nav-actions.tsx — No focus trap in mobile drawer
**WCAG:** 2.1.2 No Keyboard Trap
**Line(s):** 55–116
**Issue:** The mobile drawer opens as an overlay but does not trap focus inside. Focus can escape to background elements. The Escape key handler exists but last-focused-element is not restored on close.
**Fix:** When drawer opens, focus the first link. Trap Tab/Shift+Tab inside. Restore focus to hamburger button on close.
**Impact:** Keyboard users may tab into invisible background content while the drawer is open.

### [W5] globals.css:448 — Focus style contrast may be insufficient in dark mode
**WCAG:** 1.4.11 Non-text Contrast
**Line(s):** 453
**Issue:** The focus ring uses `var(--color-primary-subtle)` which in dark mode is `oklch(25% 0.04 195)`. Against `var(--color-bg)` at `oklch(14% 0.008 220)`, the contrast ratio needs verification. The 3px ring with oklch(25%) against oklch(14%) may fall below 3:1.
**Fix:** Ensure focus ring has 3:1 contrast against the background in both themes.
**Impact:** Low-vision users may not perceive which element has focus.

### [W6] home-page.tsx:22 — Social links section not marked as navigation
**WCAG:** 1.3.1 Info and Relationships
**Line(s):** 22–26
**Issue:** The social links are in a `<div>` but semantically they are a navigation list. Should use `<nav>` or `<ul>` with an accessible label.
**Fix:** Wrap in `<nav aria-label="Social links">` or use a list structure.
**Impact:** Screen reader users cannot identify this as a navigation region.

## Suggestion (best practices)

### [S1] nav-actions.tsx:50 — Unicode icons need `aria-hidden`
**WCAG:** Best practice
**Line(s):** 50, 109
**Issue:** The sun/moon toggle uses Unicode characters (`\u2600`, `\u263E`) as button content. While the button has `aria-label`, some screen readers may still announce the Unicode character.
**Fix:** Wrap the icon in `<span aria-hidden="true">`.
```diff
- {isDark() ? '\u2600' : '\u263E'}
+ <span aria-hidden="true">{isDark() ? '\u2600' : '\u263E'}</span>
```
**Impact:** Minor — most screen readers handle aria-label correctly, but edge cases exist.

### [S2] layout.tsx:24 — Inline dark mode script needs `<noscript>` fallback
**WCAG:** Best practice
**Line(s):** 24–29
**Issue:** The dark mode initialization script runs via JavaScript. Users with JS disabled see no theme initialization.
**Fix:** Add a `<noscript>` block that sets a default class.
**Impact:** Users with JavaScript disabled may see incorrect theme flash.

### [S3] globals.css — No keyboard focus style on links
**WCAG:** 2.4.7 Focus Visible (enhancement)
**Issue:** Links have `:hover` styles but no `:focus-visible` styles. The default browser focus ring may not have sufficient contrast.
**Fix:** Add explicit `:focus-visible` styles for links.
```css
a:focus-visible {
  outline: 2px solid var(--color-primary);
  outline-offset: 2px;
  border-radius: 2px;
}
```
**Impact:** Keyboard users may not clearly see which link has focus.

### [S4] home-page.tsx:16 — Hero bio may be empty
**WCAG:** 1.3.1 (structure)
**Issue:** The `<p>` for hero bio may render empty if `config.hero_bio` is falsy. An empty `<p>` creates unnecessary DOM nodes.
**Fix:** Conditionally render: `{config.hero_bio && <p>{config.hero_bio}</p>}` (already done — check).
**Impact:** Minor — empty elements add noise to the accessibility tree.
